Installation ^  TABLE OF CONTENTS

You will save yourself a lot of time if you follow our directions step by step.  We have MOF set up in the package to use Search and Replace to make most of the adjustments for you. In a few places you'll need to modify an actual setting.  But your best bet is to start at the beginning and let us walk you through it.  Even if you have cgi experience we can walk you through it easier than you can sort through the configuration files line by line analyzing what you need to change to install it on your site.  We know exactly where the minimum settings are to get it running in test mode on your site.

The configuration files look a little intimidating, even to some experienced cgi installers, but we will get the system up and running in test mode first, which requires only some minor adjustments.  MOF's behavior is stripped down to a very basic install when it comes out of the package, and you should not try to make behavior adjustments when you first install it.  The bulk of settings in the two MOF configuration files are used for different behavior options.  We will address that later, after you get it up and running.

 
Before you begin ^  TABLE OF CONTENTS

Note: Do not try to use Front Page or any other HTML editor to install the Merchant OrderForm scripts.  You can use your editor to work with the Example Input Pages, but not the actual script files.  

Suggestion: Do not try to install Merchant OrderForm under any SSL (Secure Sockets Layers) at first.  Get everything operating in test mode from a plain non-SSL location.  We will address using SSL as a separate installation issue.  SSL is done very different from server to server, and can cause server specific installation problems.  So, the easiest thing is to omit this for now, get it all working, then tackle transferring the back end through your SSL.  If you are not collecting credit card or checking account numbers over your web site, then you will not even be using SSL.  

If you're ready to install Merchant OrderForm on your Web Site, then we'd like to walk you through it step by step. Our instructions are designed to get the Merchant OrderForm program files, and all of our example pages, up and running on your site.  You don't have to use any of our example pages, images, or templates, but let's get them all going for you to use as a working reference. As you begin to build your own templates, product pages and images, then you can use the examples to troubleshoot your own work.  Once you have a feel for how the system works, how to build and link product input pages, then you can delete all of the packaged example pages.  We will place all of Merchant OrderForm's example pages in one directory for easy reference. You can build your product input pages wherever you like--alongside the Example Input Pages or in a separate directory.

We recommend that the zipped  package you obtained be unzipped into a working directory on your local PC, and that when you extract all the files you find the setting on whatever Zip Utility you are using to "preserver folder names."  This will place all of Merchant OrderForm's files under the same folder names they need to end up in on your server. This will help you keep all the files together.

We're going to be making some adjustments to the files before you FTP them over to your server, but here's what the directory structure should look like when you unzip the files.

  • orderform  (the package)
    • docs
    • www
      • --> cgi-bin
      • --> mof15
        • --> invoices
    • orderinfo
    • orderdata

 
1.    You need the exact absolute pathway to your domain: ^  TABLE OF CONTENTS

You need some preparatory information about your particular Web Server Account.  You may already know this information, or you may have to consult your hosting service help pages, or you may have to contact the help desk for your hosting service, but if you have this information on hand before we begin it will go much smoother. Do not guess at this part.  Guessing at it will not get things working.  

This is not a web URL, or web location address.  A web address is the URL (universal resource locator) to a web page.  What we are after does not have anything to do with where the files reside on the Web.  We are looking for the actual server pathway to the files residing in your domain space.  Servers do this differently, but on Unix-Linux you are looking for something like the following examples, which all point to the Root location of a web site:

/home/users/mysite.com/
/home/r/rga/public-web/
/u/web/ mysite/

If you are using WS-FTP then you should be able to log in to your account and see the pathway we are looking for in the right side of the screen (remote site) drop box.  This should show you where you logged into in relation to the server.  You most likely landed at the Root directory of your account space.  Get that pathway, you will need it to make Merchant OrderForm work.

On NT you are looking for something more like a Disk Location.  Here's a possible NT example:

D:\HTTPd\yoursite\

 
2.   Make directories on your server: ^  TABLE OF CONTENTS

We need to reproduce the exact directory scheme that we unzipped.  So, start up your FTP, log in to your web space, and on the Root Directory of the remote site (your server) make these exact directories:  (On Ws-Ftp use the MkDir button).

orderdata
orderinfo

Make the above directories directly off the root.  This places the directories that hold the scratch files outside any public web access.  Here's an example point of reference.  It shows where to place the directories in relation to your opening index.htm page, your .htaccess file, and your cgi-bin.

/u/web/mysite/www/index.htm
/u/web/mysite/www/.htaccess
/u/web/mysite/www/cgi-bin/your_scripts.cgi

/u/web/mysite/orderdata/
/u/web/mysite/orderinfo/

If you do not have root access to your account, then it's okay to place these two directories inside the public web access area.  These directories hold scratch files which contain cart contents and shipping address.  MOF does not store any other information to disk.  So, it is perfectly safe to put these directories anywhere, since no sensitive information is stored

Now on the remote site with your FTP proceed into your actual public web access area.  This can be named differently between servers, but usually it's something like: Htdocs, www, public-web, maybe even yourdomainname.com.  You'll know you're in the public web area when you see the cgi-bin directory.  Now make one more directory in your public web area called:

mof15

Here's an example point of reference:

/u/web/mysite/orderdata/
/u/web/mysite/orderinfo/

/u/web/mysite/www/mof15/

Then proceed into the mof15 directory, and make a subordinate directory there called:

invoices

Here's an example point of reference:

/u/web/mysite/orderdata/
/u/web/mysite/orderinfo/

/u/web/mysite/www/mof15/
/u/web/mysite/www/mof15/invoices/

Now you should have a directory structure on your server similar to what is on your local PC when you unzipped the package.  The www directory may be named differently on your server.  Important: Don't rename what the public web directory is called on your server.  Your server is configured to recognize only a specifically named directory where your Web files will reside. 

The cgi-bin directory should already be in your public web area.  It is sometimes named cgi-local.  Use the name on your server.  If it isn't there contact the help desk of your hosting service and have them set it up.  It may not work if you simply create it.  The web server software has to be configured to execute scripts in that directory.  This is especially true with NT machines.

 
3.   Adjust directory permissions on the server: ^  TABLE OF CONTENTS

Now you need to adjust the permissions (Unix-Linux) on these new directories so the server will allow the Merchant OrderForm scripts to use them to write files to.  The simplest way to do this is to use an FTP program that has a CHMOD feature, or permissions feature.  If you're using Ws-Ftp: line up the two directories orderdata and orderinfo in the right remote site window.  Right click on the folder icon, and select the chmod(unix) function. Select all the radio buttons to make these folders readable, executable, and writable for everyone (owner, group, other).  If you telnet there, then you know what to do: Chmod dirname 777.

Now, do the same thing with the mof15 and invoices folder.  Make them both readable, writable, executable by owner, group, and other.  This will make our new directories ready to be used by MOF.

Note: Some Unix-Linux servers handle permissions differently.  Some large hosting services require you to use a Web based control panel.  In some rare cases, servers can be set to run scripts under the users permissions, and they don't want you to adjust the permissions to any files yourself.  Please consult your hosting help pages or call the help desk to get more information about how your server handles permissions.  Most servers will follow the guidelines that we outlines above.

5   Adjust the URL addressing in configuration files ^  TABLE OF CONTENTS

Now we'll open up both configuration files again and do a Search and Replace on each file to adjust the necessary settings that have to do with all the URL locations of images and hyperlinks.  The URL (Universal Resource Locator) address is where files reside in relation to the Web.  This is a very different address than the absolute location of files on the server we were working with above in Step 4.  The URL is the address used by the Web to find a file,  whereas the absolute address is how the actual operating system finds a file on its hard drives.

Important: Do not use relative addressing for the URL Replace With string.  It will go much smoother if you use the complete and Full URL -- the address that a Web browser on the Internet must use to find your domain.

Do a careful Search and Replace on both mof15.conf and mofpayment.conf
Search for: http://localhost/
Replace with: http://www.yourdomain.com/

Be careful to type in the exact Search For string, ending the string with the forward backslash.  And also make sure to end the Replace With string with the forward backslash.  If you do not include these endings correctly, then the URL will not line up correctly, and MOF will not be able to find the files it will be looking for.  Make sure to Replace With only the Root URL location to your domain. 

You should have done 21 replacements on mof15.conf and 17 replacements on mofpayment.conf

 
6.   One Final adjustment to both configuration files ^  TABLE OF CONTENTS

Put in your merchant email address for error notices in both configuration files

In mof15.conf around line 185 find this line:
 $merchantmail = 'merchant@yourdomain.com';

Then in mofpayment.conf around line 163 find this line:
 $merchantmail = 'merchant@yourdomain.com';

Be careful not to accidentally erase or alter the surrounding single quotes or the ending semicolon on these lines.  Just carefully change the actual email address.  Do not change the surrounding single quotes to double quotes.  Single quotes and double quotes have different meanings in Perl.

Put in an email address that MOF will display to notify you in case a customer encounters any errors using the system.  If any errors are encountered, Merchant OrderForm will display an error message, which will be useful to you when testing your installation, and also useful for your customers to know who to report any future errors to.

8.  Set permissions on CGI files ^  TABLE OF CONTENTS

At this point you should have all necessary adjustments completed in both configuration files, and you should have all of the actual program scripts, and program support files uploaded to your server's cgi-bin

On most Unix-Linux type servers you should only need to set permissions on 3 of the above files.  We need to give both of the program files with the cgi extension full read and execute permissions for owner, group, and other.

If you're using Ws-Ftp then find and select both of the program cgi files on the right remote site window.  You are looking for mof.cgi and mofpayment.cgi   Right click on those files to get the context menu and select the chmod(unix) function on the menu.  Enable all of the boxes for Read and Execute, and only the Owner box for Write permissions.  If you are there via telnet then you know what to do: chmod filename 755.  Make sure you adjust both of the cgi program files this way.

Now you need to adjust one file with complete Read, Write, and Execute permissions.  Find the mofnumber.dat file and select the chmod(unix) function.  Enable all 9 of the boxes for complete Read, Write, and Execute permissions.  MOF must be able to write to this file.  If you're using telnet: chmod filename 777.

Okay, that should do it.  Merchant OrderForm should be installed in test mode. Easy enough. Now move to the testing section below.  When you move to test your installation of the program scripts, if you get an Internal 500 Server error, then the first thing you should do is go back to this permission process and set chmod 755 permissions on ALL of the other MOF files in your cgi-bin.  Usually you only need to adjust permissions on the three files mentioned above, but some servers may require you to set execute permissions on the others as well.  Normally those other files are not executable by the server.  The two main MOF cgi program files have their own way of executing all the support files and shouldn't need the server to execute them.

 
9.  Test the main program files ^  TABLE OF CONTENTS

At this point the Merchant OrderForm system should be installed and ready to go to work for you.  To run a simple test on it to see if it is installed properly, simply call up each one of the processing scripts in your Web browser.  You'll need to enter the complete URL to mof.cgi and then mofpayment.cgi as they are on your server.  Don't try to use any of the example input pages yet to test things. We don't want to send anything to the system,  just call up the system on your browser.

To test mof.cgi goto
http://www.yourdomain.com/cgi-bin/mof.cgi

This is the Front End cart process. It manages the cart and shipping information.  If it is operating correctly then you should get an Error Message from the MOF system informing you of a "Data Processing Error"  -- Unable to determine input mode.  Congratulations !  It's working correctly, telling you it has a problem.  Good.

To test mofpayment.cgi goto:
http://www.yourdomain.com/cgi-bin/mofpayment.cgi

The is the Back End Checkout or payment processing program.  It takes over when someone clicks the Checkout function on their cart and leads them through the payment process.  Make a direct call to that script as in the above example.  Do not attempt to send anything to it.  If it is operating correctly then you should get the Merchant OrderForm v1.53 Copyright Notice displayed in your browser.  Congratulations !  It's working correctly.  You're getting that message because it was called in an abnormal way -- it was not used by the Front End Cart, which is what it is expecting.  It will not behave that way when the actual cart is used.  It will only behave that way when someone is attempting to use it in an abnormal way.

Use the examples from our demo site to see how this is supposed to work:

  • http://www.io.com/~rga/cgi-bin/mofpayment.cgi

What if I don't have root access to my account ? ^  TABLE OF CONTENTS

Only the data and info directories should be installed at the root level of your domain.   If you don't have root access to your account, then just install the /orderdata/ and /orderinfo/ directories anywhere you want.  Keep the directories separate and don't use them for anything else.  Merchant OrderForm uses these directories to write and read scratch files that contain information about someone's cart and their shipping information.  The shipping information is needed to compute a Preview Invoice. 

 If you do not use any of the shipping, insurance, or coupon features, and don't require any computations for the Preview Invoice, then MOF will not use the /orderinfo/ directory.  You don't even need it.

Wherever you place these directories, make sure you give them full 777 permissions so the scripts can read and write to them

 
How to make your own templates ^  TABLE OF CONTENTS

Merchant OrderForm coordinates all of it's output with 7 template files.  These 7 template files are used to surround all of MOF's output so the output can be enclosed in a design that fits your site layout.  The templates are regular html pages and can be designed any way you want.  You can build or modify these templates with your favorite HTML editor.  When building these templates keep in mind that Mof will insert all of it's output at a designated point in the template page.  So you don't want to crowd up the template file, but make it with plenty of middle space to house Mof's output.

When Mof begins to return pages from it's functions, it grabs the appropriate template for that function, and prints out the template file, with it's function output inserted wherever the <!--MOF V1.5 INPUT AREA--> comment marker appears in your html template page.  Make sure this comment marker is embedded in your template where you want Mof to insert it's results.  The marker must appear exactly as you specify in both of the configuration files.  The distribution package has the marker set this way: 

$insertion_marker = '<!--MOF V1.5 INPUT AREA-->';

Some advice on making templates

It's probably a good rule to keep the templates simple.  Remember that Mof will read your template files line for line, consolidate it's output with the file, and spit back a complete html page to the browser via the cgi gateway.  So you won't be able to include anything like SSI calls in the templates.  The cgi gateway will not process SSI directives.  Also, you don't want to place anything in your templates that would trigger a reload or OnLoad() scripting directive.  You cannot reload a cgi script from the browser unless you do it manually from the refresh button.  Cgi scripts are looking for input, and an OnLoad() script directive in your template will trigger an OnLoad Event if the browser is resized.  This results in an error message because nothing was submitted to the mof scripts, so Mof will send an error message.  There should be only one way to trigger a reload of the script output, and that is when someone clicks the reload or refresh button on their browser.  Only then, does the browser know that it's supposed to submit any input that is in cache from the script.

The OnLoad() Events scripted into web pages are often used with a dynamic java script menu system.  You can use java script or fancy stuff, but just be careful.  Make sure any fancy Java Script or anything else will be compatible processing through the cgi gateway.  You can use CSS style sheets, images, links, and all kinds of other stuff in your templates.  Java script will process via the cgi gateway fine, as long as it doesn't trigger reload events.  

When building your templates use the Full URL for any links, images, CSS, etc.  The cgi gateway looks for this stuff from a  position relative to the cgi-bin, which is not necessarily the same position as if you were calling up the template file via the world wide web.  The surest fix to any of these types of problems with your templates, is to simply use the Full URL for any objects the page needs.  

If you are using SSL, then your templates will need to reference only objects via the HTTPS protocol, otherwise, a user will get the browser message "viewing non secure items over a secure page."  We'll cover more about that in the SSL section.

Security Issues ^  TABLE OF CONTENTS

Merchant OrderForm has precautions built in to prevent its program flow from being used by anyone to damage your web site, and it does not store any credit card or checking account numbers on the server.  The only information Mof stores on the server are scratch files that contain a cart's products and the shipping destination.  It can be customized to store any information you want, but unless this is programmed in, then Mof will not store any logs with important information.  So unless you re-programmed Mof, then it is impossible for anyone to get credit card or checking account numbers from any files on your server.

Important: Installing Merchant OrderForm on your server in a secure way is your responsibility.  While the program itself has been carefully thought through to prevent any internal security holes, it is possible to create security holes with a faulty installation.  Faulty installations are by far the most common security holes found when running cgi scripts on a web site.  Faulty installations are responsible for more security holes in cgi scripts than all of the fanfare about encryption.

Note: You have little security to be worried about if you are only using Merchant OrderForm to Mail or Fax payments, or with a feature like PayPal.  In the PayPal feature, all security and sensitive information is handled by the PayPal site.  Merchant OrderForm simply passes the customer off to the PayPal site to finalize payment.  If this is the case you won't be encoding or mailing any cc or check account number.

 
Is your cgi-bin secure ? Very Important ^  TABLE OF CONTENTS

What are the problems ?

The most common security hole is running your cgi scripts on a poorly configured web server.  A well configured web server will not allow any outside access to your cgi-bin from the world wide web except to execute specified scripts.  A poorly configured web server's cgi-bin will execute the specified script file if called from your browser, but will also allow any other file in your cgi-bin to be called by the browser.  So, with a poorly configured web server's cgi-bin, anyone on the world wide web could pull up a standard *.txt file, or even the *.conf or *.pl files for Merchant OrderForm and see them front and center in their browser.

This would be a serious security hole if you are using the encoding settings in Merchant OrderForm.  One would only need to call up your mofpayment.conf file and they could see from anywhere in the world what your exact encoding numbers were.  If you're not using the encoding settings, then it couldn't really cause any harm if someone saw your configuration files; however, you generally don't want someone having access to these files.

A well configured server's cgi-bin would not allow that.  If you attempted to call up the mofpayment.conf file on a properly configured virtual domain's cgi-bin then you would simply get error messages from the server saying you don't have permission to access that file, etc.

Poorly configured cgi-bins also allow a browser to list the contents of the cgi-bin directory.  You generally don't want anyone listing all the files in that directory.  A properly designed cgi-bin will prohibit a browser from calling up the contents of your cgi-bin directory and all the files living there.

You should run tests to see if your cgi-bin passes as well configured. 

Try to access the *.conf or *.pl files in your cgi-bin by a direct browser request
Example: http://www.yourdomain.com/cgi-bin/mof15.conf

Try to list the contents of your cgi-bin by a direct browser request
Example: http://www.yourdomain.com/cgi-bin/

If your cgi-bin passes these tests, then you have a well configured cgi-bin and it is secure.  We do not mean secure with regard to encryption, but secure in that no one can access files in your cgi-bin they are not allowed to.

Here are examples from two different demo sites:
Bad: http://www.io.com/~rga/cgi-bin/mofpayment.conf
 Good: http://www.merchantorderform/cgi-bin/mofpayment.conf

What are the solutions ?

Most virtual domain accounts with a reputable hosting service have properly designed cgi-bins.  If you are not on a virtual domain, but running out of a subordinate domain set up, then you run into more risk of an improperly configured cgi-bin.  The best solution is to get a virtual domain with a host that has properly configured cgi-bins.

If you don't have a properly configured virtual domain account, or can't afford one, then there are some work arounds, but they are not as good as the properly configured cgi-bin.

1.  Place a plain index.htm file in your cgi-bin. Give it a message like "no access to this area." If someone attempts to list the contents of your cgi-bin from the world wide web then they will get the no access message.  This strategy prevents a browser from listing the files in your cgi-bin.  It's also a good idea to do with all of your directories on your web site.

This measure only prevents a directory listing.  One could still make a direct call to the exact file they wanted and would be able to get it.  That would be easy to do once someone saw you were running Merchant OrderForm on your site.  Anyone can read the docs and see what the configuration files are.  So they simply ask for that file.

To prevent an exact file listing you can use these solutions:

2.  Rename both of Mof's configuration files (*.conf) to a complex name that someone will not be able to guess.  Example: make your mofpayment.conf file into: ab-999-vbxvbs-3212. You don't even have to put an extension on it.  It would be pretty hard for someone to make a direct browser request for that file.  But make sure solution number 1 is in place, else they could simply list the contents of your cgi-bin and see your new file name.

Important: If you rename your *.conf files associated with Mof scripts, then you must adjust each of the two scripts to be able to find the files by the new name you gave them.

In both of the main program scripts,  mof.cgi and mofpayment.cgi there are a few lines directly below all the copyright and disclaimer notices.  These settings in each of the two main program files must be changed to match your new file names or Mof will not be able to find the files it needs to run.  The settings you need to adjust look like this:

# DO NOT CHANGE ANY OF THESE SETTINGS
# ===================================
require 5.001;
require 'mofpayment.conf';
require 'mofpayment_lib.pl';

3.  A more complicated, yet more secure solution, is to re-configure the Merchant OrderForm system where both configuration files and both library files run as cgi scripts--meaning that we change their behavior from a *.conf and *.pl file to *.cgi files.  To make this work you'll need to alter a few lines in both configuration files and both library files.

As they come in the distribution package the two configuration files and two library files will not run as script files, but are used by the script files to load configurations and sub routines.  To make them act like script files, so the server executes them, instead of sends contents to the browser, we need to do the following:

First: rename the files
 file rename to
  mof15.conf  mof15.cgi
  mof15_lib.pl  mof15_lib.cgi
  
  mofpayment.conf  mofpayment15.cgi
  mofpayment_lib.pl  mofpayment_lib.cgi

Second: Place the Unix-Linux Perl executable identifier as the first line in each of these file.
#!/usr/bin/perl

This is the standard line, but some servers may have it defined differently. It tells the server where Perl is when the file is executed.

Third: Remove the very last line in each of those files.  The last line is simply a 1 and semicolon.  It must be removed to run as a cgi script.

Fourth: The two main program files must now have their require settings adjusted to match the new file names you just created.  The require settings will be looking for a something. cgi instead of a something. conf or something. pl.

 
The basics of SSL (Secure Sockets Layers) ^  TABLE OF CONTENTS

SSL is set up many ways on servers, and so can cause lots of variation when installing Merchant OrderForm under an SSL connection.  Here are some basic questions if you need, then we will go on to address some of the various situations one might find themselves in when using SSL and Merchant OrderForm.

Does Merchant OrderForm come with SSL ?

No.  SSL is a feature of a web server's software.  SSL is a standard part of most e-Commerce hosting packages.  It is generally not a feature of cgi scripts themselves.  You should obtain this feature as part of your virtual domain hosting package.

What is SSL ?

SSL (Secure Sockets Layers) is an encryption scheme acting between your web browser and the web server your browser is communicating with.  Once an SSL session is started on your browser, then everything between the server and your browser is encrypted, making it possible to send credit card numbers or other information between browser and server that is encrypted.

Do You need SSL with your installation of Merchant OrderForm ?

You only need to install Merchant OrderForm under SSL if you will be accepting live credit card numbers or live checking account numbers via the Web when Using the Payment Method area of Merchant OrderForm.  If you have not enabled Merchant OrderForm to accept this input, then you obviously do not need to encrypt the other stuff.  If you are only using a feature like PayPal Web Accept, then you won't be accepting any credit card numbers via the Mof system.  With something like PayPal the customer is passed from the Mof system to make payment via the PayPal system, and PayPal handles all numbers and security.

On some servers this means that you'll have to duplicate all images under a second directory where your HTTPS is pointing. This scenario means that your server is running scripts under the SSL by a virtual pointer to a different set of directories under your account.  The web server software has a standard HTTP virtual pointer to one set of directories, and an HTTPS virtual pointer to a completely different set of directories.  Both sets are usually off the same root account.  So, in this scenario you'll need to locate the actual files for each protocol in the appropriate set of directories.  That means all images, templates, scripts, and script support files.  You'll also need to change all object references to <img src="https://www.domain.com/ ..> , so that all objects occur under the HTTPS protocol.

You can even install the Mof Front End on one server, and the Mof Back End on a server across the world, and use split servers.  Handle this the same way as the first scenario above.  Make sure that the distant SSL server has all the Mof scripts, templates, images, and script support files installed under the SSL, and change all object references to <img src="https://www.domain.com/ ..> , again, so that all objects occur under the HTTPS protocol.

2. Expiring Mof's cart under SSL ^  TABLE OF CONTENTS

The most professional looking way to have Mof empty the cart contents once an order is finalized is to set the feature for deleting the cart in mofpayment.conf. However there are several ways to deal with emptying the cart at final invoice.

$delete_cart_final = 1;

Here's a further explanation of emptying the cart:

Merchant OrderForm has the ability to keep a visitors cart contents once they leave your site.  You can set the cart holdtime, and the visitors cart will still be there for that duration of time should they return to the site.

When the visitor completes the ordering process and finalized the invoice, then Mof needs a way of emptying the cart if it is being held as specified in the holdtime setting.  Mof has two ways of doing this: 1. Expire the cookie, and 2. go in and actually empty the cart file that the cookie is referring to.

There is an advantage to holding the cart and physically emptying it when an order is finalized -- a visitor still has those items in their cart if they failed to purchase them, but returned to your site at a later date.  And, even when a visitor finalizes an order, when the holdtime remains and the cart is emptied, then clicking on the viewcart function will give them an empty cart, instead of the nocookies message.

Enable the delete_cart_final switch in mofpayment.conf to turn on this method of emptying the cart.  If you are running the back end under shared SSL, and even some original SSL certificate sites, then you may get an error message when you enable this feature.  You could get a Mof Data Processing Error saying it "Can't Access the Data Files."  If this is the case, then it is because the two scripts are operating at different uid (User IDs), and the Front End created the order data files under it's uid, so when the back end attempts to access the files, it can't because of file permissions or access restrictions.  If this is your case, then enable the switch we have provided in mof15.conf called:

# ALLOW CART DELETE RUNNING UNDER SHARED SSL (Different UID) Unix ONLY
# ====================================================================

$set_ssl_chmod = 1;

This switch will direct the Front End script to create all the order scratch files where the Back End script can access them, and delete the cart contents.


Encrypting the mail ^  TABLE OF CONTENTS

The current distribution package does not have a plug in for sending mail via an encryption scheme like PGP.  What we have provided is a simple encoding method where the first set of four numbers and/or the second set of four number of the credit card are altered based on a setting in your configuration file.  These settings are in mofpayment.conf, close to the mail settings, and simply provide any integer from 1 - 9999 that will be subtracted from those sets of numbers on the credit card number.  The checking account number has a number that is subtracted from the overall account number.  These encoding settings are known only to the script and the Merchant designated mail address - receiving the mail.  So, you simply add your code number back into the credit card digits, or the checking account number and you will have the real number.
