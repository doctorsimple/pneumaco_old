How does Merchant OrderForm work ? ^  TABLE OF CONTENTS

  1. You build input pages made of html form input
  2. Your form sends data to the processing scripts
  3. The processing scripts sends results back to the browser

You build FORMs to collect user input and send the whole bag of collected input to a particular processing program by pressing the "submit button. The Forms collecting the user input are made in HTML or with an HTML editor. The CGI program processing the input is an independent program, separate from any HTML language, capable of processing what you sent it, and usually sending something back to the browser.

Web page forms are the beginning piece. 

Forms are built in standard HTML. You can build forms on a Web page with a standard HTML editor, like MS Front Page, or Adobe Page Mill, or you can build them with a plain text editor if you know how to write HTML pages.

Your forms collect information. You can build FORM input using a plain text box, a checkbox, a drop down box, a list box, or radio buttons.  When you build your input areas, each input device will have a NAME and VALUE pair associated with it.  So, if you place a simple text box on your page, you will assign this box a "name=bla-bla" and the "value" of what someone enters into the text box will be collected and sent to the processing script when someone click the submit button on your form. Likewise with other input devices, like a checkbox: You give it a "name=checkbox_one", and a value of "True/False" is collected depending on if the user checked it or unchecked it.

You'll need to read more about HTML, or how to construct FORMS in the help section of your favorite HTML editor to learn the specifics of using each input device you want to use.  But they all will function in the way mentioned above: by giving each input device a NAME, so the VALUE of that input device can be collected and used.

Here's how the FORMS you build will interact with CGI scripts.  In HTML you will define a whole area of your page as a "FORM" input collection area.  Within this FORM input area you will put all the input collection devices that you want. In HTML you begin the form collection area with:

<FORM .. > and you end the area with </FORM>
If you are using an HTML editor, then this is done for you under the hood.

In the beginning <FORM .. > tag you will be declaring two things:

  1. What "method" to use in submitting all your collected input data.
  2. Where to send the whole collection of input collected in your form area, "action"

So, when you set up the initial FORM area tag, you will define what to do with the collected data, where to send it.

This example says send all the input collected in my form to a program called: mof.cgi
We also have defined where mof.cgi is located on the world wide web.

<FORM method=POST action="http://www.domain.com/cgi-bin/mof.cgi">

So, your HTML form page has collected some input, and you have told the page to send everything to a processing program located at http://www.domain.com/cgi-bin/mof.cgi.  So, when the user presses the "Go" or "Submit" button on the form, everything is sent to the program you have addressed in the action attribute.

CGI programs accept what was sent,  processes it in some way, and usually sends something back to the browser that originally sent it information.  MOF is written in perl, but one could use other languages to interact with a server's CGI (Common gateway interface). The CGI gateway simply provides an interface between some program that exists on the server, and the World Wide Web. The server's web server software is responsible for managing this interface. CGI allows a Web browser to contact a program, and the program to talk back to the browser via the World Wide Web. This way you can process something in a real program, rather than have static web pages simply displaying text or animated gif images.


Building Your Online Store ^  TABLE OF CONTENTS

Merchant OrderForm was built to be able to accept almost any kind of html FORM input.  So whatever kind of form you are creative enough to build, you can probably get Mof to accept it and put it in the cart.  You can place a simple Buy Me button next to a product on your web pages, or you can go all out and put an unlimited number of user input options to define the product being ordered.  We have included some Example Input Pages for you to consult.  

The Example Input Pages are separated into the different ways that Mof is programmed to receive input.  There are some specific Rules that you must follow in constructing your forms, and each of the Example Input Pages has notes on the rules for that method and mode.  You can read the notes on the Example Input Pages and you can also consult the underlying HTML code in the page to see what is happening under the hood. 

 
Here's how it works: ^  TABLE OF CONTENTS

You build html forms, and the name=value pairs for your input tags will contain information about your product(s) submitted to the cart.  You must set up the name=value pairs in your product input with specific rules for the different types of submissions you want to make.

If you are form challenged, then you can try to copy and past from the Example Input Pages, or pull the Example Input Pages into your HTML editor and work with them that way.  You may have some problems pulling them into Front Page.  For some reason Front Page doesn't do a good job of recognizing the <form> tags already in place.  It's as if Front Page didn't build them, then it can't see them.

The next section outlines the specific rules that must be followed to build Input Pages that Mof is expecting.  The following notes have lots of language specific to building HTML FORM input.  So, if you get lost then you may need to brush up on How To build forms.  There are many good tutorials available free on the web.

 

Making the name=value pairs in your input ^  TABLE OF CONTENTS

Except for one of Mof's input modes, the product input that you build will always have " order" as the name for the product, and the value of name="order" will contain a string that defines information about a product.  The string containing the product information will have either 5 or 6 pieces of information about that product, depending on the mode you are using.  Those pieces of information are separated by a delimiter pattern:  Mof is already set to recognize the delimiter pattern of 4 minus signs.  You can change the delimiter in the configuration files if you want, but we know this one is reliable.

name="order"

(6) value=Qty---- Item----Description----Price----ShipCode----Tax
(5) value=Item----Description----Price----ShipCode----Tax

Price: is in decimal format, no currency symbols: 12.95
ShipCode: can be weight, or another flag, use plain numbers: 0.12 is 12/100 of a unit (lb or whatever)
Tax: is a simple switch: either on ( 1) or off ( 0).  Identifies an item as taxable (1) or not (0)

Note: The QUANTITYBOXES mode of input is done a little different.  the name=order must be numbered to match any other input associated with that product.  We'll cover more about that later.  For now, just note that all products, except those submitted under the Quantityboxes mode, will always be name="order"

Use only alpha numeric characters, and don't use any special or fancy characters.  You can use the underscore, single hyphen, periods, numbers, decimals, in your input value="Mof Product Information".  

  • Don't use any of these characters: ` ! @ # $ % ^ & * ( ) { } / \ < >
  • Don't use any Perl special characters, quotes, semicolon, etc.
  • Don't use the  | character
  • Don't use the double colon ::  
  • Don't use the square bracket characters [  or  ]

All these characters are reserve characters that MOF assigns special meaning to, so if you put them in your input values then you could confuse MOF.

Don't try to put in any HTML tags in your Input values.  HTML tags will not confuse MOF, but it will confuse some browsers like Netscape as MOF attempts to Pass and Parse these characters.

Keep everything on one line, correct your editor if it splits up long lines with line breaks in the html.  Some editors like to Tab enormous spaces in the source, and some like to split input value=lines with great white spaces and line breaks.  This sort of thing can mess up what you are trying to send MOF.

Understanding Mof's four input modes: ^  TABLE OF CONTENTS

Mof is expecting instructions and specific input for different types of products.  You instruct Mof how to process the input you are sending it by defining one of four input modes.  Mof processes product input differently for each of the four different input modes.

The first input mode is triggered when you send something via the Query String.  When Mof finds product information at the Query String level, then it goes to work to sort it out in the simplest way:  One product at a time under the name="order" where the value contains all 6 pieces of products information Mof is looking for.  If you attempt to submit anything else via the QueryString then Mof will discard it or give you an error.  The only other allowed piece of information that can be submitted via the QueryString is the ViewCart function.  You can call the ViewCart function by a standard hyperlink where the directive "viewcart" is submitted as the only piece of information on the QueryString.

When Mof doesn't find anything on the QueryString, then it proceeds to looking for POST input.  At the POST input level there are three modes that will be used to define how Mof processes the input you are sending it.  You will pass one of three modes as hidden input via the POST action under the name= postmode to define which of the three POST modes will be used.  Each of the three postmode(s) trigger Mof to process your product input differently, and so the Rules for that mode must be followed when you are making your product input.  You can see the notes further down with specific Rules for each mode.

If you fail to define what mode you are submitting products under, then Mof will tell you it's Unable to determine the post mode.  Mof will also try to check your product input, and in many situations tell you that you don't have all the necessary product information if all (5) or all (6) pieces of information are not accurate for a particular POST mode.

Each of the four input modes have Example Pages with notes.
Each of the four input modes have a Rules Checklist in the next section.
Here's a brief table to summarize how the four Modes differ:

  Use this type of input If you want to do this
1  Query String  Submit one product at a time to the cart
 With no user selected options for the product
 With no quantity input by user
   Query String  Trigger the ViewCart function
 standard link <a href="../mof.cgi?viewcart">
     
2  action=POST
 postmode=SINGLEPOST		  Submit one product at a time to the cart

 Defaults to 1 quantity if user input is omitted
 Ability to select main product from a DropBox

 The customer Checks all products wanted

Mode 1: Rules for Submitting from the Query String or the GET Method: ^  TABLE OF CONTENTS

See this example page for this type of input: ExampleSingleInput.html - The first examples

By using the query string, you can Enter a Single Item into the cart from anywhere on the Web Site to the MOF cart. You can submit using the Form GET tag or an <A HREF> link.

Here are some reminders to consult when building this type of input:

  • Any $ENV{'QUERY_STRING'} or GET has 6 Fields under the name=order
  • Submit 6 Fields under the value=Qty---- Item----Description----Price----ShipCode----Tax
  • You must include Plus+Seperators+In+This+Input+For+Spaces
  • No validation is needed for this type of input, there is no User Input allowed
  • You cannot use user input for this type of submission.
  • MOF is only set up to process user input from Form POST input.
  • If data is appearing incorrectly in the MOF Cart then Check and Re-check that your input is correctly formatted.
  • There is some limited built in ability for MOF to check ENV{QRY} input for appropriateness and MOF will give you an error message if the format is not followed for name=order value=6 Fields

The only submission via the Query String that is accepted other than an "order" is for the "viewcart" function. You can place this function anywhere on your site, on any page, either with an image, plain hyperlinked text, or a GET button.

Example link to trigger the viewcart function:
<a href="/cgi-bin/mof.cgi?viewcart">View My Cart </a>

The Query String must be ?viewcart exactly, and cannot have any value after it.
You cannot submit the viewcart request from a POST.

 

Mode 2: Entering a Single Item from anywhere on the Web Site to the MOF cart: ^  TABLE OF CONTENTS

Using the postmode=SINGLEPOST Method
See this example page for this type of input: ExampleSingleInput.html - The second examples

  • For the product
  • Specify postmode=SINGLEPOST as hidden input for this Form input
  • Using ENV{CONTENT} POST uses 5 Fields under name=order
  • Submit 5 Fields under the value=Item----Description----Price----ShipCode----Tax
  • You do not need to include Plus Separators in your input
  • The quantity will be separate input
  • Use name=quantity either as an input box or as hidden input
  • If you do not include quantity, then MOF will default it to 1
  • You can also use a DropBox with this mode 
    See ExampleDropBoxes.html - example 1
  • Adding user input to the product
  • You must have all user_input options name=whatever declared in the config file mof15.conf
  • Tell MOF which user_input options name=whatever options are Required Input from user in mof15.conf
  • If you don't identify which user_input name=whatever are Required then input for that option will not be validated
  • To use validation on a user_input option you must have the selected option value="" (Null) when the form starts or re-sets
  • Each individual product must have its own set of FORM tags

 

Mode 3: Making an Order Form Input Page of Checkboxes ^  TABLE OF CONTENTS

Using the postmode=CHECKBOXES Method
See this example page for this type of input: ExampleOrderForm.html - The first example

  • This is simple Check an item if you want it layout, no user_options
  • Submit postmode=CHECKBOXES as hidden input within the Form Tags

Mode 4: Making an Order Form Input Page with Quantity Boxes and User Input ^  TABLE OF CONTENTS

Using the postmode=QUANTITYBOXES Method
See this example page for this type of input: ExampleOrderForm.html - The second example

This is a more complicated way to build an OrderForm, or sections of OrderForms. It allows for whatever kind of additional options input needed from the user.  Submit postmode=QUANTITYBOXES as hidden input within the Form Tags.

  • For Each Product:
  • You must use matched naming conventions for your input for each product
  • Each product is assigned a unique name=orderN, (N is it's unique product number)
  • Submit 5 Fields under the value=Item----Description----Price----ShipCode----Tax
  • You do not need to include Plus Separators in your input
  • For each orderN the value=5 Fields of input
  • The quantity is submitted as a text box name=quantityN
  • The quantityN input is the only way MOF tells that an item is ordered
  • You could submit quantityN as hidden input, but this item would then always show up
    So you probably don't want to do that
  • You can also use a DropBox with this mode 
    See ExampleDropBoxes.html - example 4
  • Adding User Input to A Product:
  • You can add as many user_input options to the product as you want; however, you must declare the Root Name for the name=user_item in the config file list in mof15.conf
  • The actual Form POST input name=user_nameN will be identified by MOF as belonging to a product by the matching product number you use for it
  • Use only the Root Name in the config file to declare it as a user_input option
  • MOF looks for the Root Name and knows how to sort the numbering system to match products to options
  • You must have all user_input options name=whatever declared in the config file mof15.conf
  • Tell MOF which user_input options name=whatever options are Required Input from user mof15.conf
  • Again, you use only the Root Name for the name=user_input in the config file mof15.conf
  • If you don't identify which user_input name=whatever are Required then any input for that option will not be validated
  • To use validation on a user_input option you must have the selected option value="" (Null) when the form starts or re-sets

There is some very limited error checking built in and if you do not have your numbering system correct and MOF does not find appropriate product information matched to the quantityN input, then you could get a data processing error message.

Note: make sure your number system matches between products:
quantity3, order3, hatsize3, hatcolor3, etc
quantity4, order4, pictureframe4, framecolor4, printsize4, etc

Note: you do not have to keep sequential numbering. 
MOF will find your information if set up correctly, whenever any quantityN is detected > 0, meaning the customer wants the product. MOF will then begin looking for the orderN that matches, and look through the list of user_input Root Field Names for any user_inputN that matches that number product.

This is especially nice, because if you edit the line of products on a particular page, and delete product 15, then you don't need to renumber everything sequentially.  You can skip numbers.  The ExampleOrderForm.html shows numbers 1, 2, 3, 4 and 15. 

 

Other Considerations in making input ^  TABLE OF CONTENTS

You cannot mix the CheckBox postmode and the QuantityBox postmode within one form submission. It seemed redundant to allow for checkboxes And Quantity Box input to be mixed, since a quantity of 1 instructs MOF that the item is wanted for the cart, and a checkbox performs the same function. So, with QUANTITYBOXES you wouldn't need a checkbox to select the item, but user input to put in the Number of that product desired. The quantity box serves the purpose. So, for that reason, only the Quantity Box input postmode allows for User Input Options. It keeps the Check Box input OrderForm simple to plain uses.

You can Target="Another_Window" in your input, but you should use the exact scheme throughout your Web Site, since some of the Cart Functions will be corrupted if you submit a Target="Another_Window" for some items, and use the default Window for others. The Rule is that ALL submission possibilities must go to the exact same Window, whatever that window is for your Web Site.

 

How to Pass an Image to the Cart ^  TABLE OF CONTENTS

The way you send an image to the cart is to declare the IMG SRC url as the first part of the description field:
See this example page for image input: ExamplePictures.html

<input type="hidden" 
name="order2" 
value="Work Hat---- [img align=left hspace=10 src=http://localhost/mof15/workhat.gif width=60 height=60 border=0]The Industrial Work Hat----24.99----2----1">

Send the entire standard HTML IMG SRC tag as the first part of the description field in the value input, except don't use the regular html tag markers <img src=bla bla ..> and don't use any quotation marks, single or double.  Substitute the left and right square brackets for the regular html < tag identifiers >.  Mof will sort this out as an image.  You can't use any quotes as you normally do in regular HTML because when Mof passes this value through it's processes, then the browser will become confused and try to end your <name="order" value="line here " instead of here"> . You can see there are too many quotes now.

Mof simply substitutes your bracket beginning and ending tags [  and  ]  with the proper html beginning and ending tags <  and  > when the browser portion is printed, otherwise all the product input data keeps the [ brackets ] to enclose any html tags.  If you want to put some other simple html tags in the description, then Mof will do the same--turn them into real < html > tags when printing to the browser.  I wouldn't get too complicated, but you could pass other html tags this way.

Important Note: If you are installing Mof's back end under SSL, then all the image reference tags should call the SSL location of that image.  This will slow down the image appearing in the cart, because you are actually calling the image from one of your secure directories, but you must do it this way so that the final invoice (operating under SSL) will have the object referenced from the secure location.  IF you identify an image in your product input from standard HTTP locations, then when your customer gets to the final invoice, and Mof is ready to display the image, then your customer's browser will tell them that some items on the secure location are not secure.  Not a good message to show your customers.

 

How to make Price Adjusting product options ^  TABLE OF CONTENTS

You can specify that certain product option(s) input for a particular product are treated as special Price Adjusting Fields.  For any of your forms throughout your Store Front and site that you want to do this for, you must:

  1. Declare all Price Adjusting Fields throughout your site in mof15.conf
  2. Format all Price Adjusting input in a special way

You can only use Price Adjusting with the SINGLEPOST or QUANTITYBOXES modes.
See this example page for SINGLEPOST mode ExamplePriceAdjust.html
See this example page for QUANTITYBOXES mode ExamplePriceMultiple.html

You must follow all rules for each of these modes first.  Then for any field you specify in mof15.conf as Price Adjusting Mof will apply special processing rules, which means Mof will be looking for special input.  You can use Price Adjusting radio buttons, drop lists, list boxes all in combination with regular product options for the same product as we show in our SINGLEPOST example page.  You can even set up a very complicated group of products.  The product options can be mixed with regular input and price adjusting input.  The important thing is to adhere to the SINGLEPOST or QUANTITYBOXES input rules first, then format your designated Price Adjusting Fields.

A price adjusting field looks like this:
name="WhatEver" 
value="Option Description----2.99"

  • The name "WhatEver" must be declared in mof15.conf in @field_adjustments
  • This is the only way Mof knows to process the option input differently
  • The value has two elements separated by the default delimiter
  • value="Option Description----Price" where price is in two decimal format raw number only
  • A selection in an option can have no effect in price change, must use zero
  • Example: value="Option Description----0"
  • A selection in an option can increase the base price of the product, use a positive number
  • Example: value="Option Description----0.25"  (A 25 cent increase)
  • A selection in an option can decrease the base price of the product, use a negative number
  • Example: value="Option Description-----2.99"  (A $ 2.99 decrease)
  • Note: the subtract example uses 4 hyphens as delimiter and the fifth hyphen as negative


Any Field name that Mof sees declared in the  @field_adjustments array will be treated as having two elements to it.  Mof will process the second element of price accordingly, making adjustments to the base price for the product that option is attached to. When all price adjusting options for a given product are processed, then Mof will reset the cart price to the newly adjusted price.  Mof will also display any price adjustments next to the option data in the invoice description section for that product.

For the product description section of the cart to look well, you should probably include the Base Price of the initial product in the product description data.  That way a Base Price and any Adjusted pricing will show in the description details.  You don't need to include any price description for the Option input, because Mof will append the price to the Option description.

You can use the same Field Names for a regular field and a price adjusting field if you want to.  Mof will simply ignore any price adjustments in the value for a regular option that was submitted, because the regular options input does not have two elements for Mof to work with.  While this will work, it's probably a better idea to keep regular and price adjusting Field Names separate.  Once you declare a Field name as price adjusting, then it can be used that way throughout your site, even with different data as values.  Mof only needs to know that it is a special price adjusting field.
