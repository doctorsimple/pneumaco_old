Troubleshooting ^  TABLE OF CONTENTS

Please note, we have included some of the more common problems that folks have in getting Merchant OrderForm up and running.  Overwhelmingly, most questions we get for help installing Merchant OrderForm are from those inexperienced in installing CGI scripts.  Seasoned cgi installers, have made it through the learning curve, and can usually troubleshoot their installation.  Those not so experienced with cgi installations, can miss some very subtle things, and can get stuck.  The less experienced cgi installer's problems are invariably a subtle oversight with some small setting, some small procedure.  There are lots of different procedures, configuration settings, server settings, server protocols, files, images that must come together for Mof to be installed and operating flawlessly.

The distribution package has all configuration settings at a minimum, so we can eliminate some of the more complex installation issues at first.  The idea is to get Mof up and running with all the examples as they come out of the package, and then you can begin to customize your install to fit your site, one thing at a time.

We'll put a little section on the problems most folks encounter when attempting to install on an NT server, under MIIS, but you really should know what you're doing with cgi installs under NT and MIIS, before you even try to do this on NT, else you're in for a pretty good struggle.  You have to clearly know the ins and outs of how NT and MIIS handle absolute pathways and file-directory privileges, and especially how to use a sendmail compatible mailer under your MIIS so that Mof can mail your stuff.

For Unix-Linux type servers, running Apache (or anything else is fine), then the less experienced cgi installer will usually encounter three levels of problems:  

The first level of problem has to do with getting the scripts to run on the server. These type of problems cause Internal Server Errors.  This is a message from the server that the cgi program you attempted to run did not run correctly on the server.  This is not a Merchant OrderForm error.  It is your server telling you it could not run the script for one reason or another.  Follow the 500 Internal Server Error checklist until you get it right.  When both main script files are up and running in test mode as we have described in the Installation, then you are ready to tweak out all the ways that Mof will operate on your site.

The second level of problems occur after the main Mof script files are running on the server, but Mof is not operating successfully in one or more processes.  This is the number one, hands down, most common request we get from folks about installation problems.

Once Mof is running on the server, then any messages you receive in your browser that say ..

Merchant OrderForm v1.53 Data Processing Error

.. are messages from Mof to you telling you exactly where the problem is.  You must look very closely at what this message is saying to you.  This is your clue where the problem is.  Mof will tell you exactly what is is unable to do that it is trying to do.  These messages are your troubleshooting clues.  Don't fall prey to reading the first line and freaking out.  Read what it says underneath that.  More about that in the section below

The third level of problems is when Mof is running in all processes with no file access errors, but things like templates, images, buttons, etc, just aren't doing what you expected them to do.  Products are not showing up like you expected, templates don't look right, etc.  These are problems getting the layout, formatting, and product input to work as you want. 


Can't get main program files running - 500 Internal Server Error ^  TABLE OF CONTENTS

These troubleshooting checkpoints are for Unix-Linux type servers only.  Double and triple check everything carefully.  Many less experienced cgi installers overlook a very simple little procedure, like proper permissions, and seem to get stuck.  Use this as a checklist for your 500 Server Errors.  This checklist is not for NT.

If you cannot get the main program files running in simple test mode as described in the Installation section and are getting the error message from your server: 500 Internal Server Error

  1. Check and double check that you FTP transferred all script files in ASCII mode only. See Sending the Files

  2. Check and double check that you have followed the section on Setting Permissions

  3. Go back and set all of the MOF files in your cgi-bin with chmod 755 permissions Setting Permissions

  4. If on NT your cgi-bin must be ready to execute, set by whoever administers your NT account

  5. Check that you are calling the correct URL in your web browser.  Refresh your browser, delete the browser cache.

  6. Check that both main cgi system files have the correct location for Perl. See Where Is Perl

  7. It's possible the scripts are corrupted, either by a faulty download, or from editing.  See Corrupted Scripts.

  8. I have seen rare cases where the Hosting Service is running strict Unix and requires that the scripts actually be in Unix format to run.  I think concentric.net does this. So, if you're working on a Windows PC, then you'll have to obtain a plain ASCII text editor capable of SavingAs in Unix format.  SaveAs in Unix format before you FTP to your server.  This is rare, so I would try this as a last resort.

 
Merchant OrderForm v1.53 Data Processing Error ^  TABLE OF CONTENTS

If you find any of these type of errors you must troubleshoot the problem.  The most common error message is when MOF can't find files or directories, or when it can't read or write to files or directories.  The message will tell you which file it is attempting to work with and can't find or use.  You'll have to figure out why.  Most likely, you have not defined the absolute path correctly in the configuration file.  Mof will tell you what file it is trying to access, not by name but by its function:  

Unable to Locate-Make Data File: 20000909081224-3938445
Unable to Write To Orders Data File: 20000909081224-3938445

  • Mof cannot create and/or update your order data files
  • Mof cannot access the orderdata directory
  • Absolute path defined in mof15.conf for $datadirectory is not correct
  • The orderdata directory does not have full write permissions

Unable to Write To Preview Information File: 20000909081226-3938057

  • Mof cannot create and/or update your order info files
  • Mof cannot access the orderinfo directory
  • Absolute path defined in mof15.conf for $infodirectory is not correct
  • The orderinfo directory does not have full write permissions

If you're getting Unable to Read Template file errors, Mof is saying it can't get to the template files.  Either you don't have them unloaded to the server, you don't have the proper absolute pathway defined in mof15.conf for each of these templates.  As per the installation instructions, you should have uploaded all of the 7 template files and 1 no cookies file into the ../mof15/  directory. 

Here are the 7 template file error messages and corresponding setting.

 
Error Message mof15.conf setting template file
Unable to Read Template File: Main Template File $accept_order_template mofmain.html
Unable to Read Template File: Validation File $validation_template mofvalidate.html
Unable to Read Template File: Preview Information File $preview_template mofpreview.html
Unable to Read Template File: Preview Template File $preview_info_template mofinformation.html

 
Error Message mofpayment.conf setting template file
Unable to Read Template File: Payment Information File $payment_info_template mofpayinfo.html
Unable to Read Template File: Payment Final Template File $final_template mofpayfinal.html
Unable to Read Template File: Save Invoice Template File $save_invoice_template mofinvoice.html

Check to make sure the configuration settings are correct for the file MOF is attempting to access.  You'll have to track down the appropriate configuration file (Tip: which script is running ? mof.cgi or mofpayment.cgi), and the applicable configuration setting variable.  Make sure that you have the correct Absolute Pathway defined for the setting, and make sure the absolute pathway you defined in the setting is exactly where the file resides on the server and your domain file structure.  This is probably the most common installation error.

Merchant OrderForm should give you few errors outside file usage errors.  The other possible places Merchant OrderForm will give you error messages is when you start to Build your own Store Front and mess up some of the Rules for submitting a POST to the Merchant OrderForm system.  But this should not happen with the example input pages, since they are already thoroughly tested, and are set up correctly.

If you are Building your HTML Product Input Pages to POST to the Merchant OrderForm system, and getting these messages:

  • Unable to determine input mode
  • Product order data not available

Then you have errors in how you are constructing your HTML Product Input Pages.  Consult the section on Building your Online store and consult the Example Input Pages in the package.  The Example Input pages have all sorts of instructions, and hidden comments inside the HTML code.  View the HTML source for those Example pages to troubleshoot a particular method.


Formatting, Layout, and Product Input problems ^  TABLE OF CONTENTS

Once the Merchant OrderForm scripts and support files are operating correctly on your web site, then a third level of problems might be from the interaction of Product Input pages, templates, images, etc.

If your products are not entering the cart, or the cart does not see the products as you intended, the first thing you should carefully check is the exact syntax underlying your product input pages.  Consult the documentation on Building Input Pages.   

You must follow the exact rules for submitting product information to the cart.  Remember that different postmodes have different rules, and that product options must be declared in the configuration file.  Make sure if you are using postmode=QUANTITYBOXES that your product numbering scheme is accurate, and that each product's number matches any product options input number.  Make sure you are using the correct value for a particular input mode.  Some postmodes submit 5 pieces of information for the product, some submit 6.

The best thing you can do to sort out these type of errors is to gain a better understanding of how html forms work.  


Some tips for NT and MIIS ^  TABLE OF CONTENTS

Our best advice for those of you wanting to install Mof on an NT and MIIS system, is to know what you are doing.  If you are experienced with cgi Perl script installations under NT and MIIS, then you will know what to do to make it work.  If you are not experienced with NT and MIIS, then we will offer some very sparse tips:

You must realign everything to run as *.pl files.  Rename the two main script files to *.pl files instead of *.cgi files.  Most NT and MIIS servers run Perl cgi scripts under the extension PL.

Don't worry about the first line in the two main script files, Whereis Perl.  NT doesn't use this, it uses the PL extension to identify what type of file it is.

Force all Full and absolute pathways for all your file, template, and URL settings.  Don's use any relative addressing, but use the complete pathways.  

Example: refer to the processing script files with rull URL, don't take it for granted that NT will automatically understand that you want to run mof.pl from the native directory that cgi is running from.

$programfile = 'http://www.domain.net/cgi-bin/mof.pl';
$paymentfile = 'http://www.domain.net/cgi-bin/mofpayment.pl';

Again, use very complete absolute addressing for any template file locations:
Make sure you use the backslask and not the forward slash.
Make sure the setting is not enclosed in double quotes.
Using the backslash inside double quotes causes malformed addresses that don't work.

$accept_order_template = 'e:\websites\domain\mof15\mofmain.html';

If you are referencing a directory then you'll need two ending backslashes.  If you don't do this you'll get a 500 Internal Server Error.  What happens is that the backslash has powerful meaning in Perl, and you need to tell Perl that you want a backslash as an literal ending character on the pathway, else, Perl will truncate the ending quote.

$infodirectory = 'e:\websites\domain\orderinfo \\';

MIIS does not necessarily want to run cgi scripts, since that is a little outside the BG empire, and you could encounter all kinds of problems configuring MIIS to run Perl scripts.

It's possible that MIIS won't run Perl scripts because of 'require' problems.  Mof uses standard directives to require certain support files to run when it loads up.  MIIS might not recognize these support files as valid unless you declare them as valid in @INC and require them using the Full absolute pathway.

An example using the above pathways to your cgi-bin to force MIIS to run the required support files from the cgi-bin might look like this:

Declare your support files in @INC when the script starts up.
You'll need to insert lines similar to this as your script starts up.

BEGIN {
push(@INC, 'e:\websites\domain\cgi-bin\mof15.conf');
push(@INC, 'e:\websites\domain\cgi-bin\mof15_lib.pl');
}

Then all require lines will reference the full absolute address

require 'e:\websites\domain\cgi-bin\mof15.conf';
require 'e:\websites\domain\cgi-bin\mof15_lib.pl';

The rule if you're having trouble under NT and MIIS is to not leave anything for MIIS or NT to interpret, but force all settings to their absolute location, so NT doesn't have to try to understand what you want to do.

All settings that would normally run under the native cgi-bin relative location via Unix-Linus should be given full pathways for NT.  Don't allow NT to think anything out.

$use_country_list = 'e:\websites\domain\cgi-bin\countries.txt';
$use_state_list = 'e:\websites\domain\cgi-bin\states.txt';

The above examples should solve your file pathway problems, but the most common problem installing under NT and MIIS is making sure that MIIS and NT are allowing all the required file and directory access privileges.  These file and directory access privileges do not operate like on Unix-Linux.  You'll have to contact your admin or hosting help pages to determine what you need to do to get all privilages operating correctly.  Sometimes, the cgi-bin under NT systems is not even set up to execute PL files, so you'll need to contact someone to even have that enabled.

Lastly, before Mof will be able to mail you anything, you'll need to have a sendmail compatible mailer installed under the NT and MIIS system.  We'll try to construct a Mof plugin that works with SMTP directly for NT.  You can check out support pages.

 
Maintenance Issues ^  TABLE OF CONTENTS

Merchant OrderForm uses two directories to read, write, and store data for cart products and shipping information.  Mof creates a separate file for each customer, for both cart products and shipping address information.  The current version 1.53 of Mof does not have any provisions for cleaning out these scratch files.  Scratch files are made in these directories.

../orderdata/
../orderinfo/

You should periodically check these directories, sort them by date files were created or modified, and delete any files in either of those directories that are older than the corresponding cookies settings you have for the cart and shipping information.

I have several thousand files operating out of one of my directories, so you don't need to worry that you may have accumulated many files.  Unix-Linus is capable of housing many files without problems.  

You have cookie settings in mof15.conf for how long to keep a customer's cart file and shipping information file.  See the sections:

How to keep or not keep the cart if a purchase is not completed
How to keep or not keep a visitor's shipping information

Any files that are older than your cookie settings can be safely deleted.  Any scratch files that are older than the cookie settings you have will not be used by Mof because even if the customer does return to your site, their reference number for stored information has been expired as an active cookie.  So go ahead and delete any of these files.

Don't worry.  It's okay to delete all the files if you want to.  The worst that will happen is the customer will have to complete their shipping information again, or they will not have a saved cart.  Mof will know to create them a new file, even if there is a cookie reference, but their file has been deleted.

 
Where is your Perl ? ^  TABLE OF CONTENTS

The two main cgi files for Merchant OrderForm called mof.cgi and mofpayment.cgi identify where the Perl interpreter is located on your server.  This is done on the very first line of each of these script files.  They are set in the package for the standard location of Perl on most Unix-Linux type servers.  In rare cases you may need to modify this line to point to the location of the Perl interpreter if your server has it located in a different place than the one we have set:

Most Unix-Linux type servers use this location for Perl:
#!/usr/bin/perl

If your server uses a different location, then you should be certain what that location is, and replace the first line in both mof.cgi and mofpayment.cgi with your server's location of Perl.  Important: The line must be the first line in the script, and it must start with #!  

Here's an example of a different location:
#!/usr/local/bin/perl

 
Are the scripts corrupted ? ^  TABLE OF CONTENTS

If the Perl syntax has been corrupted in any of the program files:

  • 2 *.cgi files
  • 3 *.pl files
  • 2 *.conf files
  • 2 *.mail files

Then Merchant OrderForm will not operate.  You could get the server message: 500 Internal Server Error.  This is not common and you shouldn't assume this is the case unless you have reason to suspect it.  

The most common cause of corruption with these files is when you are editing them.  Beginners make mistakes, leaving off a quote, changing a single quote to double quote or visa versa, accidentally deleting the semicolon which ends every function or command.  Any erroneous alteration of the Perl language syntax within the scripts will cause the script to fail when the Perl interpreter attempts to compile or execute the script.  The language within the scripts themselves is not forgiving like html.

Unfortunately, the only way to check if the scripts are corrupted is to use the Perl interpreter to check them.  The beginner cgi installer may not have access to this or know how to call it directly.  So, if you suspect the files are corrupted then you might try downloading the package again, and moving through the initial install carefully.

If you have access to the Perl interpreter then you simply need to call a -c (code check) on the file.
 perl -c filename.cgi

This will tell you if the code is okay or what line there are syntax errors.
You can do this from Telnet, from the Unix-Linux command shell, or from a DOS shell if on NT or Winxx

 
Can't access the invoice files ^  TABLE OF CONTENTS

When running under shared SSL you may be unable to access, delete, or change the web copies of the invoice files that were created.

The Cause:

Usually the Back End final invoice script is running under shared SSL, and the server has designated all scripts running thusly as either "www" or some other UID (User ID). So when you log in under you normal FTP User ID, you don't have permissions to mess with those files.

In mofpayment.conf enable the setting in the Save Web Copy of Invoice section:

SAVE WEB COPY OF INVOICE
SAVE WEB COPY OF INVOICE

ALLOW ACCESS TO INVOICE FILES RUNNING UNDER SHARED SSL (Different UID) Unix ONLY

$set_ssl_chmod = 0;

Enable this switch to have the Mof Back End create invoice files under shared uid (user Id's).  Note this is for Unix-Linux servers only and we have not worked with this problem under NT.

 
Data Processing Error when enabling Delete Cart Final ^  TABLE OF CONTENTS

The error is usually from the same cause as the above error.  You are running under shared SSL and the Front End and Back End are running under different uid (user ID's).  When the Delete Cart Final feature is enabled, then the Back End cannot access the cart data files that were created by the Front End.  Enable this fix switch to have the Front End create shared permissions so both ends of the cart can use those files.  This is for Unix-Linux only.

In mof15.conf enable the setting in the 

COOKIE SETTINGS
COOKIE SETTINGS

ALLOW CART DELETE RUNNING UNDER SHARED SSL (Different UID) Unix ONLY

$set_ssl_chmod = 0;

 
When using SSL I get "unsecured items" messages ^  TABLE OF CONTENTS

If you have the Mof front end installed under regular non-secure HTTP protocol, and have the Mof back end installed under the secure HTTPS (SSL) protocol, then you have some meticulous double checking to do.  The rule is this: any object that could possibly come up under the SSL connection (being run via mofpayment.cgi), just have a reference to the secure HTTPS location of that object.  This means any image, cart buttons, logos, template images, images for products placed in your cart, etc.  You'll have to track down just exactly where the culprit is, which image is still being called via the non-secure HTTP URL. 

You may want to consult the sections of the Installation instructions for SSL:

  • Configuratinos for mof.cgi (mof15.conf)
  • Configurations for mofpayment.cgi (mofpayment.conf)

There are three places non-secure images could be coming from.

First, the configuration file itself.  See the section above for mofpayment.conf for a detailed list of what images are referenced by this file.  These settings will call on navigation buttons, validation cue images, the paypal button, and background images for the Ship To and Bill To tabs on the Final invoice.

Second, the two templates that the Final Processing uses to collect Billing information and display the Final Invoice could have standard references to images, rather than the SSL reference.

  • ../mof15/mofpayinfo.html
  • ../mof15/mofpayfinal.html

If you want to test to see if those two templates above are secure: Then call them up in your browser independent of the script. Just put in the Full HTTPS url to your browser address. Does the template page load without any "insecure items" messages ? If it doesn't load in the plain browser as secure, then it will not load as secure when the script uses it. Make these tests to any templates that the script will be using under the SSL.

Third, you could be using the picture function in your cart, placing images of your products into your cart, and those image src img="HTTP .." regular image references. You must reference any of these images from the beginning as under SSL, so that when you get to the Final Invoice the URL is HTTPS.  They will end up however they start out.  The Final Invoice doesn't magically know to convert how you referenced them.

Tip: To trouble shoot where the non-secure image is coming from, view the html source for the page that triggers the non-secure message. Then do a search for http, not https.  Look closely at every image reference in the source and make sure if it is an image then it is referenced using HTTP only.  Any HTTP reference for an image call will trigger the dreaded message you are so diligently trying to get rid of.  Remember: a reference in the html source may have its origins in either the configuration file, one of the template files, or even from placing pictures into your cart.

 
I can't get the mail to work ^  TABLE OF CONTENTS

There is a test file in the ../docs/ directory that came in the package.  We put it there becasue we didn't want folks uploading this file unwittingly.  If you upload the test file, then you MUST make sure to remove it when you are done running it.  If you don't then anyone that has a copy of this documentation can easily run the file.  The test file can't do anything like delete or edit files, but it can give out the absolute addressing to your site.  So make sure you unload it if you use it.  Don't leave it on the server.

The test file <mof_path_test.cgi> will test for 3 know locations of mail programs, 2 for sendmail, and one for qmail. It will tell you if any, and which ones, of those locations are good.

If you can't get the mail to work, then first make sure you have it turned on, and that you have put in good return addresses in the settings for Merchant mail.

If that is correct, then you should run the test from your server's cgi-bin or cgi-local, wherever the scripts are located, and match one of the good Mail settings to the sendmail settings in your mail plugins.

In merchant.mail and customer.mail close to the top there is a setting for where the sendmail program is: double check to see that you have the correct location for your server.  If not you'll need to assign the correct location for your mail program in this setting for both mail plugins.

$mail_program = '/usr/sbin/sendmail -t';
