About Database Integrity?

A word of caution and advice about database integrity. Once you go live with the ARES system, you should let ARES manage everything from that point on. You should not attempt to manage any of the data by editing it by hand. Always use the ARES interfaces: Either the Admin Utility or the Affiliate Sign Up and Management screens.

Data is related through all the datafiles, and you just can't go deleting an Affiliate in the infofile.dat unless you are real clear on how to delete all info related to that Affiliate in the other datafiles for passwords, coupon rates, etc. Likewise with Representatives.

If you delete or edit information by hand, then the system will not crash, you could risk getting some very unpredictable results. ARES know how to handle things. You should just let it do things. I already knows how to account for information changes, updates, transaction deletions, etc.

 

Two Important Rules

Rule Number One: Do not turn off File Locking for ARES & MOFcart scripts. By default, file locking is enabled.

Operating ARES with MOF requires that multiple files work with the same data in close range. If you do not want to risk losing your data, then you should double and triple check that All File Locking settings are enabled throughout the ARES and MOF system. All File Locking is turned on in the distribution packages.

Rule Number Two: Backup, backup, backup.

Databases are like motorcycle riders: There are two kinds: (1) Those that have been down, and (2) those that are going down. Backup all your *.dat files. If you observe Rule Number One above you can feel very confident that your data will be safe, barring any server crashes, etc. However, you should still practice good backup habits. The rule is "Backup Often, Backup Quick."

 

Email links everywhere?

The Admin Utility will attempt to hyperlink all email addresses that it finds. This is so you can click and call up your email program on the fly to send instant email messages to various Affiliates, Customers, or Reps related to the task you are doing. You'll see this at various places in the Admin Utility as you navigate through functions.

Example: You add a new Representative. On the confirmation message, click on the new Rep's email address to send an on the fly message to them. Or you change an Affiliate's earning rates (they are one of your top sellers). The confirmation message will attempt to hyperlink their email contact to send messages.

 

If you have trouble logging in

Always make sure you "Log Out" of the Admin Utility. If you close your browser without actually logging out, then you will not be able to log in under the old password any longer. You'll have to create a new password, log in and out successfully at least once. Then you can change the password back to the old one. Folks usually only forget the Log Out touchiness once.

 

Zero out all datafiles before you go live

We recommend you get everything thoroughly tested to your satisfaction and then completely zero out the data files and start over with clean datafiles.

Warning: You should not attempt to manage the *.dat files manually, deleting individual records, because all 6 *.dat files have relational connections, and data is stored across all 6 files for a given Affiliate. The best way to preserve relational integrity among the data files is to let ARES manage everything. Once you are tested and ready to go live, simply zero out ALL the files and start over.

The only file that you could safely remove data from is the transaction log - activitylog.dat. You can remove individual lines in this file safely. You are permanently removing a transaction when you do this.

Backup all your datafiles regularly, and Insure that all File Locking settings are enabled on completely in the ARES and MOF scripts.

 

Using the search boxes

The search boxes str will allow you to search for that pattern in all fields for a Rep or Affiliate record. You can use a period for a wildcard. The pattern is not case sensitive. The search will search All fields for a Rep or Affiliate to match on.

Tip: If you enter one or two periods into the search str box, then ARES will list all available records. You probably don't want to do this with more than 100 or 200 affiliates, since the request will return a very long list to your browser. But if you have under a few hundred Affiliates in your database, then it's an easy way to list them all, see when you have new ones, etc.

Note: The setting below limits the number of records that a search will return. To have the Admin Utility return all records or more records, you'll need to narrow your search or increase the Allowed number of recrods that can be returned in the following setting:

  • $NarrowCouponSearch = nn;

Note: Your pattern or string must be found contained within a field. In other words: A search for "Belinda Williams" will yield a No Records Found message because "Belinda" is in the First Name Field, and "Williams" is in the Last Name Field. Hence, the pattern is not found contained in any field. The pattern really exists in two separate fields.

Just search for "Belinda" and you'll get where you need to go.

 

How to List new Affiliates that signed up?

The Admin Utility doesn't have a specific function to do this, however, here's some search tips that can help you accomplish it.

Use a search pattern that will match on the Affiliate membership Start Date field. The membership start date is always formatted with leading zeros this way mm/dd/yyyy.

Example: match all records for Feb 2001: str: "02/../2001"
Example: match all new affiliates signued up Feb 14, 2001: str: "02/14/2001"

We don't have date functions installed so you can't search by greater or less than. You're limited to searching for a specific day, or for the month. You can search by the month, and send out personal greetings to each new affiliate from the search results page.

 

How do you delete Affiliate-Reps from the database

You don't. The Admin Utility can delete or void transactions only. The only reason you would delete a transaction is if it was a charge back, or the payment was not fulfilled for some reason, and you don't want to pay out earnings on an invoice that you never collected on.

Once an Affiliate or Rep has been entered into the database, then you don't want to delete them because they could have associated transactions, or a Rep could have Affiliate associations. ARES still needs all this information to create reports and stats.

To nullify a particular Affiliate: Find their record and set both customer discount rates, and their earning rate to zero. Any further transactions logged under their Affiliate Number will be zero earning transactions. When you go to list all "unpaid earnings" the nullified Affiliate may show transaction records, but with no earnings. You can safely "pay" the zero based earnings and that will update the transaction log.

To nullify a particular Representative: Find their record and set their earning rate to zero. Also, conduct a search of Affiliate Records for that Rep Number, and edit each of the associated Affiliates to have a different Rep associated with them.

So, instead of thinking in terms of deleting Reps or Affiliates, think in terms of nullifying their earnings to your system, or configuring their settings in such a way as to nullify their activity.

Another Trick to limit or nullify a particular Affiliate or Representative's activity in your system is to assign a specific redirect URL to that Affiliate, redirecting anyone coming from their site (Affiliate Link) to a particular message on your site.

Another Trick is to set the Override Previous Cookie to ON in the ares.cgi script. Then assign a different Full Affiliate Link as a redirect URL for the nullified Affiliate.

  • http://www.yousite.com/mcart/ares.cgi?ID=NEWNUMBER

This effectively assigns a new Affiliate Number to all click throughs from the nullified Affiliate Link, stopping transaction activity for that Affiliate, and redirecting it to someone else. Be careful not to put the system into an endless loop. If you accidentally assign the nullified Affiliate's redirect URL with their own Affiliate Number, then you could create an endless loop.

 

For Maximum Benefit Become A Representative

We recommend you place yourself in the ARES system, both as a representative and as an Affiliate.  You should be the first to register.   See more about this in the Operating Documentation. You can have all Rates for Customer Discounts, Affiliate earnings, and Rep earnings set to zero if you want. We suggest this because ARES can become a nice little way to analyze all your site's buying activity, but you need to have a default Rep and Affiliate for customers who don't arrive at your site from a real world Affiliate Link.

Setting up a default Rep and Affiliate allows you to keep All transactions made on your site for mailing lists, etc. If you do not plan on using Reps, or the first level commissions, then you will want to make yourself the sole Rep in the database. By doing this, you'll be able to create a complete report on all buying activity on your site, with a nice analysis grid of all Affiliate Activity. You can get a one page summary of Affiliate performance for all Affiliates in your system.

If you use both tiers, and have real world Reps you are using in the system, then you'll only be able to create these reports per each Rep. You can see how each Rep's set of Affiliates are performing.

 

When do newly adjusted rates become effective?

The very next transaction following a rate adjustment from the Admin Utility will reflect the new rates. Rate adjustments are not retroactive. If an Affiliate or Rep has pending transactions that were legitimately earned at an agreed upon rate, then their earnings remain.

You can adjust Customer Discount Rates, Affiliate Earning Rates, Representative Earning Rates as often as you want, but any transactions that are posted are posted under the rates existing at the time it posts.

 

Adding A New Representative to the Database

Representatives are not allowed to Sign Up online as Affiliates are. Only the Administrator can add new Reps to the database. You can only add new Reps via the Admin Utility. The Admin Utility will assign the next available Rep Number based on what is already assigned in the database, incrementing the highest number found by the increment setting:

$RepCodeIncrement = 100;

The Admin Utility also can be set to a default Rep Starting Earnings Rate.

$RepStartRate = 0;

If you have Require Reps enabled for New Affiliate Sign Ups (signup.cgi), then a Rep is required for Each Affiliate who signs up. You'll have to have at least one Rep entered into the system for this to work. You can find more info about this in the Operating Documentation: Setting up the Sign Up process

The New Rep is immediately available to the Affiliate Sign Up process as soon as you add them via the Admin Utility.

When you have successfully added a new Rep to the database, the screen will appear back at the LOGGED IN: ADMIN MENU. A confirmation message will appear top screen to tell you they were added successfully.

Added New Representative Successfully - Rep # xxx

From the ADDING NEW REPRESENTATIVE screen, you can Log Out or return to the main Menu and the record will not be saved. The only way to save the record is to click on the "Add New Rep" button below the Rep information input. That screen will prompt you if any information is incomplete. You always have the choice of Logging Out or returning to the main Menu from an incomplete information screen and the New Rep record will not be saved.

View-Manage Account Activity for a Rep

From the REPRESENTATIVE SEARCH RESULTS screen, click on the View function right screen by the Rep's record you want to Manage Account for. This takes you to the REPRESENTATIVE ACCOUNT HISTORY screen. All transaction data is accessible from this screen, and you will be able to:

  • Get a summary of all transaction activity - Account Summary
  • Get a quick count of Affiliates currently assigned to this Rep
  • List all Pending payments (earnings) for this Rep - Payment Pending Detail
  • List a History of all past payments to this Rep - Previously Paid Detail
  • Get an analysis of all Affiliate-Customer activity for this Rep Affiliate Stats

Tip: When Viewing Rep transactions, you can filter what is displayed by the checkboxes for pending, history, affiliate stats (top screen). The default info that is displayed can be set in the uadmin.cgi settings for:

  • $view_rep_pending = 1;
  • $view_rep_history = 0;
  • $view_affiliate_stats = 1;

Here's detailed information on the what you can do on the REPRESENTATIVE ACCOUNT HISTORY screen:

Account Summary

The Account Summary always displays. It summarizes Paid and Not Yet Paid transactions for the Rep collected in the transaction data file.

Payment Pending Detail

This is where you can View Pending Earnings and/or generate a Statement of payment for a Rep. Use this to select what transactions you want to pay the Rep for, and click the "pay" arrow at bottom right of the list. The transaction log will be updated to reflect your selections and you will get a Statement detail which you can Print or SaveAs to mail the Rep along with your company check for that amount. In Internet Explorer you could even > Select > Copy and > Paste into an email message HTML format.

All transactions in the database for that Rep which you have not yet posted as paid will be listed here. By default, all of these transactions are selected. You can deselect any transactions that you do not want to include in this payment update.

Very important: If you use this list to pay Reps, then you should Print or SaveAs the STATEMENT presented. This is the only chance you will get to print or save this statement. It is not a recorded statement, but a copy of the updated activity for that Rep that you just directed ARES to do. When you leave the statement page the nice little summary is gone, and the information becomes a part of the "Previously Paid Detail."

Previously Paid Detail

This is a record of all previous activity you posted as paid for that Rep. Records are sorted by transaction date, most recent date to top of list. If you want to print a historical record, then select only the "history" checkbox top, and submit by yellow arrow. You can then Print or SaveAs a historical record for that Rep.

Affiliate Stats

This will show you a summary grid of all Affiliate performance associated with that Rep. The summary will list all preserved data in the transaction log. So, even though you may have assigned a particular Affiliate to another Rep, if that Affiliate had transactions in the past while associated with the current Rep, their stats will show, but only for that Rep. The summary is only for buying activity that made it into the transaction logs. If an Affiliate is not listed here, they have not been credited with any sales.

Note: If you are not using the Rep level for commissions, but have made yourself the only Rep, then this function will give you a powerful one page listing of all Affiliate performance for your system. If you have set a default Affiliate as yourself, then you will have an entire site summary of how your orders are progressing--what sales are from Affiliates, what sales aren't from Affiliates.

Records are sorted by top earners. This is a fast way for you to see who is selling, click on that Affiliate's email address and send a quick Thank You note, or a note that you've upgraded their commission rates.

You'll also see some quick Affiliate Stats upper right of the REPRESENTATIVE ACCOUNT HISTORY screen. Here's what the numbers and amounts mean.

  • Assigned Affiliates = Number of currently assigned Affiliates
  • Earning Affiliates = Number of any Affiliates with earnings under this Rep
    ( current or not current )

  • Earnings   Total earnings in dB: Pending and Previously Paid
  • Payments Total payments you posted to this Rep
  • Pending   Total Pending earnings for this Rep

Assign or Change an Affiliates Redirect URL

From the AFFILIATE SEARCH RESULTS screen, click on the URL function right screen by the Affiliate's record you want to assign or change a redirect URL for. This takes you to the AFFILIATE REDIRECT URL screen. The information on that screen will tell you the current status of the Affiliate: if they have a redirect URL or not, and what it is if they do.

Affiliates by default do not have a redirect URL associated with them. When Affiliates sign up this is not an option for them to select. If you have a special reason for re directing a specific Affiliate's click throughs to a certain URL on your site, then you can set the URL using this function.

ARES will direct you to either "Add" or "Change" an Affiliate's URL. The URL is not added or changed until you click the "Add" or "Change" button lower center screen, and receive the confirmation message on the LOGGED IN: ADMIN MENU screen:

Added Redirect URL Successfully for # Number

The URL will be hyperlinked so go ahead and test it to see that it is leading to the correct place. When a redirect URL is set for an Affiliate, then all visitors arriving at your site via the Affiliate's Link will end up at the page on the ARES confirmation message. They will go where you went when you click on the confirmation message URL. So test it, then use your browser back button to return to ARES

If an Affiliate does not have a redirect URL listed for them, then the system will send all click throughs from that Affiliate to the Default URL defined in ares.cgi

  • $defaulturl = 'http://www.yourplace.com/shop/';

How it works: When a visitor clicks on the Affiliate Link, they are sent to ares.cgi, which sets the cookie, and then checks the URL redirect information to see if that particular Affiliate has a special redirect page. If one is found, then ares.cgi will redirect to that page, otherwise, ares.cgi will redirect to the default page.

If you want to remove an Affiliate's redirect URL, then pull up their record, select URL, and delete the existing redirect URL listed for them in the text box. Click on "Change." The redirect URL for that Affiliate was deleted when you return to the LOGGED IN: ADMIN MENU screen and receive this message:

Deleted Redirect URL Successfully for # Number
Affiliate does not have a current Redirect URL, defaults apply

How do I void an invoice?

The "Transaction Management" section has this function. You'll need the exact Invoice Number to enter into the "Find" Box. You can't enter a part of the Invoice Number and get search results. You should know the exact Invoice Number or you wouldn't be trying to void it.

Enter the Invoice Number into the "Find" Box. If the invoice is found, you'll be presented with a "Delete Invoice #####" button. You'll also see a message to the right of the button about what you are getting ready to delete. The message will confirm the Invoice Number, Date, Amount, and Affiliate Number associated to the transaction. Check that it is correct, then click the "Delete Invoice ######" button.

Nothing will be deleted unless you click the "Delete Invoice #####" button. It is safe to click on any other button on the page at this point and the screen will refresh without your "Find" Invoice Number to delete.

If you click the "Delete Invoice #####" button, the Invoice will be deleted from the transaction log as if it never existed. You will return to the LOGGED IN: ADMIN MENU and a confirmation message will appear stating:

Successfully Deleted Invoice # Number

Warning: The invoice will be gone as if it had never existed. So if you delete an invoice that has already been included in creating an Affiliate or Rep STATEMENT, then the information will be gone. It will not be listed any longer in the Rep's or Affiliates Account History. Be careful to pay Rep or Affiliate earnings only on invoices where the customer's payment has cleared.

Make Mailing Lists

You can create an ASCII text file containing email addresses from the Make Mailing List function. Sorry, we did not include automated sendmail abilities in this feature because if the lists are very long, then you wouldn't want to bulk mail from the browser this way. So, we only created the ability to create the list.

Give a name to the File that you want to create. The file will be created in the directory you have defined in uadmin.cgi settings for:

  • $mail_file_path = '/ABSOLUTE_PATH_TO_DATAFILES/';

Select the options you want from the right area, and click the "List" button. A file will be created in the pathway above containing the email addresses that match your options selected. If the file already exists you will get a "File Already Exists" message. This is to prevent you from overwriting any important files on your server.

  • Email addresses for customers come from the entire transaction log
  • Email addresses for Affiliates come from the Affiliate database
  • Email addresses for Reps come from the Rep database
