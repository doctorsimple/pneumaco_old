Welcome 

The ARES v2.5 system is fully independent and runs on flatfile databases which require no special web site modules or databases.  It is fully contained and manages all aspects of operation from Affiliate sign up to Admin control via the web.  Once installed, everything is done via the web browser.  It has been tested to 10,000 affiliates, but is should easily run up to 150,000 affiliates.

After Installation Important Reminders

Please read this section. It has important information about some of the message adjustments. ARES is packaged for easy installation, however, some of the settings and information may not be what you want for your particular Affiliate program.

1. The signup.cgi script is preset to issue all New Affiliate signups a 10% commission rate, and their referred customers a 10% discount on an order made through your site. If you will not offer a discount to customers, or will offer a different rate, etc. then you should make the correct settings in your signup.cgi script before you go live. Discounts and earning rates will be applied with any live orders placed once the system is turned on. Make sure you're not discounting products 10% unless you really want that. This is just a starting Sign Up rate which ARES issues to a New Affiliate. The Admin is able to adjust discounts and rates at any point, but it's possible that a New Affiliate could sign up and send someone through for the discount quick. So, don't set a starting discount or commission unless it's for real

Find all the Settings for Affiliate Sign Up in the /mcart/<signup.cgi> script. You should be familiar with editing Ascii files and downloading/uploading to the hosting server.

2. Review and make any adjustments to the content in the Operating Agreement and the F.A.Q., especially regarding your terms, rates, payment intervals, and policies.

These are plain HTML pages located here:

http://www.gardenthemes.com/ares/agreement.html
http://www.gardenthemes.com/ares/faq.html


3. Make any adjustments to the Sign Up confirmation messages that the signup.cgi script will display for a Successful Registration, and email confirmation. Make sure the message matches what you say in your Operating Agreement and F.A.Q.

You can see more about that toward the end of this document in the section Adjusting Automated Sign Up Messages.

Yourself as Representative and Default Affiliate

Affiliates Sign Up process

All affiliates will Sign Up at a common location on your site.  They will use the form in this package called: signup.html.  You can construct the page any way you like to match your site, but all the <FORM></FORM> components must remain, i.e., the form field names.

Point the signup.html form to the signup.cgi script, as it is in the default installation

http://www.gardenthemes.com/ares/signup.html

If you are using the two tier level: Representatives, you have the option to set the signup.cgi script to require a Representative.  

# REQUIRE A REP TO BE SELECTED ? 

$require_rep = 1;

If you enable this setting, then all new sign ups must have a representative associated with their application.  The system provides the following ways to assign a representative to the new affiliate.

  1. Place a hidden field in the signup.html 
    This will default to a particular representative for all new Sign Ups.
     <input type="hidden" name="Ecom_Rep_Number" value="100">
  2. Place a text box for the rep number on the signup.html
    The affiliate can enter the rep number with other signup info
    <input name="Ecom_Rep_Number" size=8>
  3. If the rep number is not provided from the signup.html, either as hidden or text input, AND you have enabled require_rep in the signup.cgi, then the system will present a list of available Representatives and require the new affiliate to select one before they can complete the sign up.

Important Note:  You must have at least one representative entered into the system before you can enable the require_rep setting.

You can also set the starting customer discount rates and starting Affiliate earnings rates in the signup.cgi configurations. A new Affiliate will be started at these designated rates, but you can change their earnings rate or their associated Customer Discount Rates at any point via the Admin Utility.

$discount = 0.05;
$affiliate_rate = 0.05;

If you are using the require Rep switch above, then you have the option of allowing the Rep's contact information to be viewable to the Affiliate on emails, account management, etc. Set to zero to show only the Rep's email addr, city, and state. Set this to one to show full contact information, including address and phone number.

$allow_rep_info = 1;

How Are Referrals Tracked ?

Whenever a site visitor clicks into your site from an Affiliate Link, then the script ares.cgi sets a cookie on the visitors browser, looks for any redirect information specific to that Affiliate, and redirects to the appropriate page on your site. The visitor only sees the redirect page and usually never knows they had a cookie set, unless they have message boxes turned on for their browser monitoring cookie activity. The cookie is extremely harmless, and only holds the Affiliate's Number.

When the same visitor returns to your site within the specified time frame and completes a purchase via the Merchant OrderForm Shopping Cart, then MOF will check for any Affiliate Numbers, and if one is found, an affiliate transaction is logged into the ARES system, according to the Affiliates settings: customer discounts, Affiliate rates of earnings, Representative associations and credits, etc.

Some of the various settings in <ares.cgi>

  • @NOT_ALLOWED_DOMAINS = ();
    Prevent certain referring domains from using the system

  • $holdcookie = 720;
    Number of hours to hold Affiliate Number as cookie

  • $override_previous_cookie = 0;
    This will allow the most recent Affiliate's click through to get credit

  • $allow_cookie_to_expire = 0;
    Prevents re setting the time interval for same customer
    The cookie only lives as long as the time set at first click through

  • $defaulturl = 'http://www.yourplace.com';
    Where to send visitor by default if no redirect info found


Notes: When ARES & MOFcart are running together you can only use ARES for the Second Discount (The Coupon Discount cannot be used while ARES is enabled). The Primary Discount can still be used with either/or ARES & Coupon Discount. In other words, you can use either of the scenarios below, but not both at the same time.

1. A Coupon Discount independent of the ARES system, where you can issue Coupon Numbers and customers can manually enter that Coupon Number into the Shipping Information page of MOF. This number is stored along with all other Shipping, Insurance, information.

2. The ARES ver 2.5, which is fully automated, and uses cookies to track click-throughs from Affiliate Links. If ARES is enabled the Affiliate Number will not be stored in the MOF shipping information file, so there is never any chance of using a stored Coupon Number.

Very Important: The ares.cgi script must be located in the exact same directory as your mof.cgi script is. This is so that Merchant OrderForm can extract the cookie set by ARES, by using the exact same path.

Setting Allowed Domains in the scripts

You can set "Allowed Domains" in both the signup.cgi and usrmgt.cgi scripts. This prevents any referring URLs not listed as "allowed" from POSTing data to your scripts. The setting is an array. Include all possible domains to be allowed, including aliases. The installation search and replace should have created your root domain as the only allowed domain.

@ALLOWED_DOMAINS = ('http://www.youplace.com');

Here's how you would add more domains to the list.

@ALLOWED_DOMAINS = (
'http://www.youplace.com',
'http://youplace.com',
'https://www.youplace.com',
'http://IP NUMBER'
);

You must get the usage of single quotes and commas exactly correct. The last item on the list is not followed by a comma, but all items before it are. All items are surrounded in single quotes.

Adjusting automated Sign Up messages

You'll want to make sure your Operating Agreement (agreement.html), F.A.Q. (faq.html), and the automated messages generated by the Sign Up script (signup.cgi) are all in agreement about your terms and conditions, i.,e., how you will pay out earnings to Affiliates. They are all in agreement in the package, but if you changed the terms that define how you will pay, then you'll have to adjust the message in all three places above.

The installation package reads this way in all three places:

Referral revenue is paid to you monthly by the 10th of the following month. Accounts with a balance less than $25.00 will be carried over to the following pay period.

You can adjust the automated registration confirmation message that appears in the browser when an Affiliate has Successfully completed the online Sign Up process, by going into the signup.cgi script, to the first section right after the End Configurations comments.

# END OF CONFIGURATIONS
# END OF CONFIGURATIONS

# EDIT THE "HOW IT WORKS" HTML MESSAGE HERE
# EDIT THE "HOW IT WORKS" HTML MESSAGE HERE

$HowItWorksMessage = "

You can put any plain html here between these markers
as we have it in the package. Find your way to the line that explains
the terms for Referral revenue to be paid.

"; # do not remove this line

To adjust the similar message appearing in the automated email confirmation that an Affiliate receives after they have successfully completed the Sign up process, continue down in the same section until you see the commented area:

# EDIT THE "HOW IT WORKS" MAIL MESSAGE HERE
# EDIT THE "HOW IT WORKS" MAIL MESSAGE HERE

$HowItWorksMailMessage = "

Adjust your message here using any plain text as we have it.
Do not use HTML. This is an email message.

"; # do not remove this line

Note: when you are inserting text in these areas you are writing inside Perl syntax. Anything between the double quotes will be printed; however, you cannot use special characters like the dollar sign unless you precede the character with the \$ back slash. Example: to say $ 25.00 you must use this syntax: \$ 25.00.

Using screen templates in ARES scripts

In the package, we have provided a template for headers and footers which the scripts for Sign Up <signup.cgi> and Account Log In <usrmgt.cgi> use to mix with their dynamic output. The template matches our packaged example for the Associates Are pages.

You can, of course, use any template you like, but you must have the needed template insertion marker in the template as in our example:

In <signup.cgi> and <usrmgt.cgi> these settings define what template to use

$template = '/home/gardenth/public_html/ares/temp_referral.html';

$insertion_marker = '<!--INSERT_TEMPLATE_OUTPUT-->';

Once you have defined the template use this way, then you will need to have that exact Insertion Marker in the template, at the point you want ARES to mix its output with the template file, usually in the middle, between the header and footer. Make sure the Insertion marker appears on a line completely by itself. It's a good idea to duplicate the Insertion Marker as we did in the package example. This will make sure the marker is found by ARES.

You should keep the templates free of any fancy JScript or especially VBScript, and most especially free of any onLoad() or event scripting. A plain html page is recommended. Fancy Scripting can cause problems when the stuff attempts to run from the CGI environment.
